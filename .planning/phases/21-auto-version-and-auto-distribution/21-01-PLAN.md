Replace the current `release` job with:

```yaml
  release:
    needs: [build-macos, build-windows, build-linux-amd64, build-linux-arm64]
    runs-on: ubuntu-22.04
    permissions:
      contents: write
    steps:
      - name: Download all artifacts
        uses: actions/download-artifact@95815c38cf2ff2164869cbab79da8d1f422bc89e  # v4.2.1
        with:
          path: artifacts/

      - name: List artifacts
        run: ls -laR artifacts/

      - name: Get version from tag
        id: version
        run: echo "VERSION=${GITHUB_REF#refs/tags/}" >> $GITHUB_OUTPUT

      - name: Upload artifacts and publish release
        uses: softprops/action-gh-release@153bb8e04406b158c6c84fc1615b65b24149a1fe  # v2.6.1
        with:
          tag_name: ${{ steps.version.outputs.VERSION }}
          draft: false
          files: artifacts/**/*
```

Key changes per D-09, D-10:
1. ADD `permissions: contents: write` to the release job (needed for updating the release)
2. ADD "Get version from tag" step that extracts the tag name from GITHUB_REF
3. REPLACE "Create GitHub Release" step with "Upload artifacts and publish release" step
4. CHANGE `draft: true` to `draft: false` — this publishes the release, triggering distribute.yml per D-10, D-11
5. ADD `tag_name: ${{ steps.version.outputs.VERSION }}` — tells softprops to find the EXISTING release by tag (created by release-please) instead of creating a new one
6. REMOVE `generate_release_notes: true` — release-please already wrote release notes from CHANGELOG.md; keeping this would overwrite them (Pitfall 5 from research)

The `softprops/action-gh-release` SHA remains `153bb8e04406b158c6c84fc1615b65b24149a1fe` (v2.6.1, already pinned). This version correctly detects and updates existing releases when tag_name matches.

Do NOT modify any build jobs. Do NOT change the workflow trigger (`on: push: tags: ['v*.*.*']`). Do NOT add `generate_release_notes: true`.
