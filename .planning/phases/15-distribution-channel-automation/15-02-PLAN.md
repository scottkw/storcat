<task type="checkpoint:human-action" gate="blocking">
  <name>Task 1: Create GitHub repository secrets for distribution PATs</name>
  <read_first>
    .github/workflows/distribute.yml
  </read_first>
  <action>
    The user must create two GitHub repository secrets for the distribute.yml workflow:

    **Secret 1: HOMEBREW_TAP_TOKEN**
    - Type: Classic PAT (or fine-grained with contents:write on scottkw/homebrew-storcat)
    - Required scope: `repo` (classic) or `contents:write` on `scottkw/homebrew-storcat` (fine-grained)
    - Purpose: Allows the `update-homebrew` job to push the updated cask file to `scottkw/homebrew-storcat`
    - Where to create the PAT: GitHub Settings -> Developer Settings -> Personal access tokens
    - Where to add the secret: GitHub -> scottkw/storcat -> Settings -> Secrets and variables -> Actions -> New repository secret
    - Name: `HOMEBREW_TAP_TOKEN`
    - Value: the generated PAT

    **Secret 2: WINGET_TOKEN**
    - Type: Classic PAT only (fine-grained tokens do NOT work with winget-releaser/Komac)
    - Required scope: `public_repo`
    - Purpose: Allows the `update-winget` job to fork microsoft/winget-pkgs and submit a PR
    - Where to create the PAT: GitHub Settings -> Developer Settings -> Personal access tokens -> Tokens (classic)
    - Where to add the secret: GitHub -> scottkw/storcat -> Settings -> Secrets and variables -> Actions -> New repository secret
    - Name: `WINGET_TOKEN`
    - Value: the generated classic PAT
  </action>
  <how-to-verify>
    After creating both secrets, verify they appear in:
    GitHub -> scottkw/storcat -> Settings -> Secrets and variables -> Actions

    Both `HOMEBREW_TAP_TOKEN` and `WINGET_TOKEN` should be listed.
  </how-to-verify>
  <resume-signal>Type "secrets created" when both HOMEBREW_TAP_TOKEN and WINGET_TOKEN are added to the storcat repository secrets</resume-signal>
  <verify>
    <automated>gh secret list --repo scottkw/storcat 2>/dev/null | grep -q "HOMEBREW_TAP_TOKEN" && gh secret list --repo scottkw/storcat 2>/dev/null | grep -q "WINGET_TOKEN" && echo "PASS" || echo "FAIL - secrets not found (may need gh auth or manual verification)"</automated>
  </verify>
  <acceptance_criteria>
    - `gh secret list --repo scottkw/storcat` output includes HOMEBREW_TAP_TOKEN
    - `gh secret list --repo scottkw/storcat` output includes WINGET_TOKEN
  </acceptance_criteria>
  <done>Both HOMEBREW_TAP_TOKEN and WINGET_TOKEN secrets exist in the scottkw/storcat repository.</done>
</task>

<task type="checkpoint:human-action" gate="non-blocking">
  <name>Task 2: Submit first WinGet package to microsoft/winget-pkgs</name>
  <read_first>
    packaging/winget/manifests/s/scottkw/StorCat/2.1.0/scottkw.StorCat.yaml,
    packaging/winget/manifests/s/scottkw/StorCat/2.1.0/scottkw.StorCat.installer.yaml,
    packaging/winget/manifests/s/scottkw/StorCat/2.1.0/scottkw.StorCat.locale.en-US.yaml,
    .planning/phases/15-distribution-channel-automation/15-RESEARCH.md
  </read_first>
  <action>
    The winget-releaser action requires that `scottkw.StorCat` already exists in microsoft/winget-pkgs before it can auto-submit updates. The first submission must be done manually.

    This task is NON-BLOCKING because:
    - The automation code is complete regardless
    - The Homebrew channel works independently
    - The first submission can be done with v2.2.0 (the first release built by the new pipeline) rather than the 2.1.0 stubs
    - WinGet PR review takes days/weeks -- cannot block phase completion

    **When a release is published and the real installer SHA256 is known:**
    1. Update the 2.1.0 stub manifests (or create new version manifests) with the real SHA256
    2. Fork microsoft/winget-pkgs
    3. Create a branch with the manifests under `manifests/s/scottkw/StorCat/{version}/`
    4. Submit a PR following the winget-pkgs contribution guidelines
    5. Wait for automated validation and maintainer review

    **Alternative:** Use `komac` CLI locally:
    ```bash
    # Install komac
    brew install komac  # or cargo install komac

    # Submit (will prompt for GitHub token)
    komac update scottkw.StorCat --version {VERSION} --urls "https://github.com/scottkw/storcat/releases/download/v{VERSION}/StorCat-v{VERSION}-windows-amd64-installer.exe" --submit
    ```

    Once the first version is accepted into winget-pkgs, subsequent releases will be auto-submitted by the distribute.yml `update-winget` job.
  </action>
  <how-to-verify>
    Search for the package in microsoft/winget-pkgs:
    ```
    gh search prs --repo microsoft/winget-pkgs "scottkw.StorCat" --state open
    ```
    Or check if already merged:
    ```
    gh api "repos/microsoft/winget-pkgs/contents/manifests/s/scottkw/StorCat" 2>/dev/null && echo "EXISTS" || echo "NOT YET"
    ```
  </how-to-verify>
  <resume-signal>Type "winget submitted" when the first PR has been submitted to microsoft/winget-pkgs (does not need to be merged yet)</resume-signal>
  <verify>
    <automated>gh api 'repos/microsoft/winget-pkgs/contents/manifests/s/scottkw/StorCat' 2>/dev/null && echo 'PASS - package exists' || echo 'INFO - not yet submitted (non-blocking)'</automated>
  </verify>
  <acceptance_criteria>

  </acceptance_criteria>
  <done>First WinGet package submission PR created in microsoft/winget-pkgs (or acknowledged as pending until first real release).</done>
