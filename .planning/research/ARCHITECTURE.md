# Architecture Research: CLI Integration for Go/Wails Desktop App

**Domain:** Go/Wails v2 desktop application — adding CLI subcommands to existing GUI binary
**Researched:** 2026-03-26
**Confidence:** HIGH — based on direct source code analysis + Wails v2 docs + community findings

---

## Scope

This document focuses on the v2.1.0 question: how do CLI subcommands integrate with the existing StorCat Go/Wails architecture without breaking the GUI path? The v2.0.0 architecture (current state) is documented here as the baseline, followed by the recommended integration design.

---

## Current Architecture (v2.0.0 Baseline)

```
main.go
  └── wails.Run(Bind: []*App)
        └── App (app.go)
              ├── catalogService  → internal/catalog/service.go
              ├── searchService   → internal/search/service.go
              └── configManager   → internal/config/config.go

frontend/wailsjs/go/main/App.js   (auto-generated by wails dev/build)
frontend/wailsjs/go/models.ts     (auto-generated)
frontend/src/services/wailsAPI.ts (hand-written compatibility shim)
```

### Existing Component Responsibilities

| Component | Package | Responsibility |
|-----------|---------|---------------|
| `App` | `main` | Wails-bound API surface — all public methods callable from frontend |
| `catalog.Service` | `internal/catalog` | Directory traversal, JSON + HTML catalog generation |
| `search.Service` | `internal/search` | JSON catalog reading, recursive search, browse metadata |
| `config.Manager` | `internal/config` | Config JSON read/write at `~/.config/storcat/config.json` |
| `models` | `pkg/models` | Shared data types — pure data, no logic |

### Isolation Rules (Must Remain Intact)

- `internal/*` packages do NOT import each other
- `internal/*` packages do NOT import the Wails runtime
- `pkg/models` has no logic — data shapes only
- `App` is the only coordinator; services never call each other

## CLI Integration Architecture (v2.1.0 Target)

### Core Decision: os.Args Dispatch in main.go

The entry point `main.go` inspects `os.Args` before calling `wails.Run()`. If a subcommand is present, run CLI logic and exit. If no args, launch the GUI as today. This is the only correct pattern — do not attempt to run CLI commands through the Wails GUI path.

```
storcat               →  wails.Run()  →  GUI launches
storcat create ...    →  cli.Run()    →  executes, os.Exit(0)
storcat search ...    →  cli.Run()    →  executes, os.Exit(0)
storcat version       →  cli.Run()    →  executes, os.Exit(0)
```

**Why this works:** `wails.Run()` is never called for CLI invocations. The Wails framework is inert — it contributes embed.FS overhead but no runtime cost. Services (`catalog.Service`, `search.Service`) are plain Go structs with no Wails dependency; they work identically from CLI context.

**Critical production build finding:** `wails build` compiles with `-tags desktop,production`. The `production` tag is what gates the `//go:embed all:frontend/dist` asset embedding. CLI-only code paths that return before `wails.Run()` are fully compatible with production builds — the embed.FS is linked in but never accessed in CLI mode. There is no build tag conflict.

## Pattern 1: main.go Dispatch

**What:** `main.go` decides GUI vs CLI before any framework initialises.

**Why not a separate `cli` package import of `cobra`:** Six subcommands do not justify the cobra dependency. The Go standard library's `os.Args` check plus a simple dispatch table is sufficient and adds zero dependencies. Cobra adds ~2MB to binary size and pulls in `pflag` — unnecessary overhead for a desktop app where binary size is a selling point.

**Recommended implementation:**

```go
// main.go
func main() {
    // CLI mode: any known subcommand as first arg
    if len(os.Args) > 1 {
        switch os.Args[1] {
        case "create", "search", "list", "show", "open", "version", "help", "--help", "-h":
            os.Exit(cli.Run(os.Args[1:]))
        }
    }

    // GUI mode (no args or unrecognised args fall through)
    runGUI()
}

func runGUI() {
    app := NewApp()
    cfg := app.GetConfig()
    // ... wails.Run(...) as today
}
```

**The `cli.Run()` signature:**

```go
// cli/cli.go
func Run(args []string) int {
    // args is os.Args[1:] — subcommand is args[0]
    // Returns exit code: 0 = success, 1 = error, 2 = usage error
}
```

**Why `int` return, not `error`:** CLI commands print their own errors to stderr and control their own exit code. Returning `int` lets `main.go` call `os.Exit()` cleanly without triggering deferred cleanup that Wails might register. Returning error and calling `os.Exit` from main is equivalent but less clear.

## Pattern 2: Service Instantiation in CLI Context

**What:** CLI commands construct services directly — same constructors as the GUI path, no shared singleton.

**Why not reuse `App` struct:** `App` has a Wails context (`ctx context.Context`) that is only valid during `wails.Run()`. In CLI mode there is no Wails runtime context. Using `App` from CLI would require nil-checking `app.ctx` everywhere and is architecturally wrong — `App` is the Wails binding layer, not a general-purpose service facade.

**Correct pattern:**

```go
// cli/create.go
func runCreate(args []string) int {
    // Parse flags from args
    fs := flag.NewFlagSet("create", flag.ContinueOnError)
    title := fs.String("title", "", "catalog title")
    output := fs.String("output", "", "output name (without extension)")
    // ...
    if err := fs.Parse(args); err != nil {
        fmt.Fprintln(os.Stderr, err)
        return 2
    }

    // Instantiate services directly — same as NewApp() does
    svc := catalog.NewService()
    result, err := svc.CreateCatalog(*title, dir, *output, "", nil)
    if err != nil {
        fmt.Fprintf(os.Stderr, "error: %v\n", err)
        return 1
    }

    fmt.Printf("Created: %s (%d files)\n", result.JsonPath, result.FileCount)
    return 0
}
```

**Services are already CLI-compatible:** `catalog.Service` and `search.Service` are empty structs with no Wails imports. They use only `os`, `filepath`, `encoding/json`, and `github.com/djherbis/times`. No changes needed.

## Pattern 3: Shared Helpers via pkg/cli (Optional)

If multiple commands share output formatting logic (table rendering, byte formatting, tree printing), extract it to `pkg/cli/output.go`. Do not put this in `internal/` — the `internal` convention implies it is not imported by sibling packages. `pkg/` is the right home for shared utilities.

```
pkg/
├── models/catalog.go      (existing)
└── cli/                   (new, optional)
    └── output.go          print helpers: table, tree, bytes
```

This is only needed if two or more commands share non-trivial output code. For v2.1.0, inline formatting in each command file first; extract only when duplication becomes real.

## Pattern 4: Version Access in CLI

`version.go` uses `//go:embed wails.json` which is a `main` package file. The `Version` variable is accessible from `cli/` because `cli` is a separate package — it cannot access `main.Version` directly.

**Solution:** Move version to `pkg/version/version.go` or pass it as a parameter to `cli.Run()`.

Recommended approach — pass as parameter:

```go
// main.go
os.Exit(cli.Run(os.Args[1:], Version))

// cli/cli.go
func Run(args []string, version string) int { ... }
```

This keeps `//go:embed wails.json` in `main` (correct — `wails.json` is at the root, and embed paths are relative to the file), avoids a new package, and makes the CLI testable without embed machinery.

## Pattern 5: Windows Console Output (Critical Platform Concern)

**Problem:** Wails production builds on Windows use `-H windowsgui` (via `-ldflags`), which detaches the process from the console subsystem. When a user runs `storcat create ...` from a Windows terminal, stdout and stderr are silently swallowed — the command appears to do nothing.

**Finding:** This is a documented limitation of Wails v2 on Windows (GitHub issue #544). The Wails maintainer confirmed that having both GUI (no console flash on double-click) and CLI (stdout visible in terminal) simultaneously is architecturally challenging due to Windows' subsystem model.

**Recommended mitigation:**

Use `AttachConsole(-1)` via `syscall` at the top of `cli.Run()` on Windows. This re-attaches the process to the parent console after the fact. Combined with redirecting `os.Stdout`/`os.Stderr` to the console, output becomes visible.

```go
// cli/cli_windows.go  (build-tag-scoped to Windows)
import "syscall"

func attachConsole() {
    syscall.NewProc("AttachConsole").Call(^uintptr(0))  // -1 = attach to parent
}
```

Call `attachConsole()` at the top of `cli.Run()`. This is a well-known Windows pattern; Daniel Tillett documented it in 2013 and it remains valid.

On macOS and Linux, terminal I/O works normally — no special handling needed.

**Alternative if AttachConsole is unacceptable:** Provide a separate `storcat-cli` binary for Windows (same codebase, `go build` without Wails tags, no GUI assets). This is simpler but requires users to install a second binary. Not recommended for v2.1.0.

## Pattern 6: wails build Compatibility

`wails build` passes `-tags desktop,production`. The `production` tag is what activates the `//go:embed all:frontend/dist` line in `main.go`. The `desktop` tag gates some Wails platform-specific code.

**CLI code in `cli/` has no build tag constraints.** It compiles under all tags because it imports only standard library packages and internal services that also have no build tags.

**The only constraint:** Do not put `//go:embed` in `cli/` files. Embed directives in non-`main` packages must use paths relative to their own directory. Frontend assets live at project root in `frontend/dist`, not relative to `cli/`. Leave all embeds in `main.go`/`version.go`.

**Manual build for pure CLI binary (for testing/distribution):**

```bash
go build -o storcat-cli ./  # without wails tags — no WebView2, no frontend assets
```

This produces a CLI-only binary without the GUI stack. Useful for testing CLI commands in isolation without the full Wails build pipeline.

## Data Flow: CLI Command Paths

### storcat create

```
os.Args = ["storcat", "create", "--title", "My Docs", "/path/to/dir"]
  → main.go: switch "create" → cli.Run(["create", ...])
    → cli/create.go: flag.Parse(args[1:])
      → catalog.NewService()
        → svc.CreateCatalog(title, dir, output, "", nil)
          → traverseDirectory() → *models.CatalogItem tree
          → writeJSONFile() → dir/output.json
          → writeHTMLFile() → dir/output.html
        return: (*models.CreateCatalogResult, nil)
      → fmt.Printf("Created: %s (%d files, %s)\n", ...)
    return: 0
  → os.Exit(0)
```

### storcat search

```
os.Args = ["storcat", "search", "report", "/path/to/catalogs"]
  → cli/search.go: flag.Parse(args[1:])
    → search.NewService()
      → svc.SearchCatalogs(term, dir)
        → os.ReadDir() for .json files
        → searchInCatalogFile() per file
        return: []*models.SearchResult
      → print table of results to stdout
    return: 0
  → os.Exit(0)
```

### storcat list

```
os.Args = ["storcat", "list", "/path/to/catalogs"]
  → cli/list.go
    → search.NewService()
      → svc.BrowseCatalogs(dir)
        return: []*models.CatalogMetadata
      → print table: name, title, size, modified
    return: 0
```

### storcat show

```
os.Args = ["storcat", "show", "/path/to/catalog.json"]
  → cli/show.go
    → search.NewService()
      → svc.LoadCatalog(filePath)
        return: *models.CatalogItem
      → render tree to stdout (reuse or re-implement generateTreeStructure for terminal)
    return: 0
```

**Note:** `catalog.Service.generateTreeStructure` renders HTML (uses `&nbsp;`, `<br>`, `html.EscapeString`). The CLI `show` command needs a plain-text tree renderer. Either add a `GenerateTextTree(item) string` method to `catalog.Service` (exported, reusable) or implement it inline in `cli/show.go`. Prefer the exported method — it makes the service more useful and keeps `cli/` thin.

### storcat open

```
os.Args = ["storcat", "open", "/path/to/catalog.json"]
  → cli/open.go
    → derive htmlPath: replace .json → .html
    → os.Stat(htmlPath) to verify existence
    → platform open: exec.Command("open", htmlPath)     # macOS
                      exec.Command("xdg-open", htmlPath) # Linux
                      exec.Command("cmd", "/c", "start", htmlPath) # Windows
    return: 0
```

**Do NOT use `runtime.BrowserOpenURL` from Wails** — that requires an active Wails context. Use `os/exec` directly. The `github.com/pkg/browser` package (already in go.sum as a Wails transitive dep) is a cleaner option.

## Build Order for Implementation

Dependencies flow from bottom (shared) to top (entry point). Build in this order:

### Step 1: catalog.Service — Add Text Tree Renderer (5 min)

Add `GenerateTextTree(item *models.CatalogItem) string` to `internal/catalog/service.go`. This is independent of everything else and unblocks `cli/show.go`.

The existing `generateTreeStructure` is HTML-specific (uses `&nbsp;`, `<br>`, `html.EscapeString`). The text version uses plain `├──` / `└──` connectors and `\n`.

### Step 2: cli/ Package — Core Scaffolding (15 min)

Create `cli/cli.go` with the `Run()` dispatch function and help text. No service calls yet — just routing and `--help` output. This lets you verify the dispatch structure before implementing commands.

### Step 3: cli/version.go and cli/create.go (30 min)

`version` is trivial (one line of output). `create` exercises the full `catalog.Service` path — implement it second to validate service instantiation from CLI context.

### Step 4: cli/search.go and cli/list.go (30 min)

Both use `search.Service`. Implement together since they share the same service constructor pattern.

### Step 5: cli/show.go (20 min)

Depends on Step 1 (text tree renderer) and the `search.Service.LoadCatalog` method.

### Step 6: cli/open.go (20 min)

Platform-specific `exec.Command` for browser open. Test on macOS first (`open` command), note the Windows/Linux variants.

### Step 7: cli/cli_windows.go — AttachConsole (20 min)

Add the Windows console attachment shim. Can be developed and tested on Windows only; the file is silently ignored on macOS/Linux via the filename convention.

### Step 8: main.go — Wire Dispatch (10 min)

Add the `os.Args` switch and the call to `cli.Run()`. This is last because all CLI commands must compile cleanly first.

**Total estimated implementation: ~2.5 hours of focused Go work.**

## Anti-Patterns to Avoid

### Anti-Pattern 1: Using App Struct from CLI

**What people do:** Instantiate `NewApp()` in CLI commands and call `app.CreateCatalog(...)`.

**Why it's wrong:** `App.startup()` expects a Wails context. `App.SelectDirectory()` calls `runtime.OpenDirectoryDialog(a.ctx, ...)` which panics with nil ctx. The App struct is the Wails boundary — it exists to bridge Go services to the Wails JavaScript runtime. Using it outside that context is misuse.

**Do this instead:** Instantiate services directly: `svc := catalog.NewService()`.

### Anti-Pattern 2: Calling wails.Run() for CLI Mode with Hidden Window

**What people do:** Launch `wails.Run()` with an invisible window, route CLI output through the Wails event system, then exit.

**Why it's wrong:** `wails.Run()` initialises the OS WebView, loads the embedded frontend, and blocks. On macOS it creates a menu bar entry. On Windows it shows a taskbar icon. The startup time (~300ms for WebView init) is visible. Output must go through channels/events rather than stdout. This is massive complexity for zero benefit.

**Do this instead:** `return` before `wails.Run()` for CLI paths.

### Anti-Pattern 3: Cobra for Six Simple Subcommands

**What people do:** Add `github.com/spf13/cobra` as a dependency for CLI subcommand routing.

**Why it's wrong for this project:** Cobra adds ~2MB binary overhead via its `pflag` dependency. StorCat's selling point is small binaries (93% smaller than Electron). Six subcommands with simple flag sets are trivially implemented with `flag.FlagSet` from stdlib. Cobra's auto-help, shell completion, and middleware are not needed here.

**Do this instead:** Use `flag.NewFlagSet(name, flag.ContinueOnError)` per subcommand. Write help text as a `const`. This is 20 extra lines per command versus cobra's zero, but keeps the binary lean.

**Caveat:** If requirements grow to 20+ subcommands with nested hierarchies, revisit cobra. The threshold is real command complexity, not raw count.

### Anti-Pattern 4: Importing Wails Runtime in cli/ Package

**What people do:** Use `runtime.BrowserOpenURL` in `cli/open.go` since it's already in go.mod.

**Why it's wrong:** `runtime.BrowserOpenURL` requires a Wails context (`a.ctx`) which is nil in CLI mode. It will panic. The `pkg/browser` package (already a transitive dep via Wails) provides platform-aware browser opening without a Wails context. Use that.

**Do this instead:** `browser.OpenURL(htmlPath)` from `github.com/pkg/browser`.

## Binary Size Impact

The `cli/` package adds pure Go code with no new external dependencies (using only stdlib + existing transitive deps). Binary size increase is negligible — estimated 10-30KB of compiled Go code for six simple commands.

The Wails framework code, WebView2 loader, and embedded frontend assets are linked in regardless. They are unused in CLI mode but cannot be excluded without a fundamentally different build. The `wails build` toolchain does not support conditional asset embedding based on runtime arguments. This is acceptable — the CLI binary is the same binary as the GUI binary.
