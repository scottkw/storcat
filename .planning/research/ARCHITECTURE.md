# Architecture Research: CI/CD Pipeline and Repo Consolidation

**Domain:** GitHub Actions release pipeline + installer packaging + cross-repo automation for Go/Wails desktop app
**Researched:** 2026-03-27
**Confidence:** HIGH — based on direct source analysis (existing build.yml, scripts, wails.json) + verified GitHub Actions Marketplace docs + Wails documentation

---

## Scope

This document covers v2.2.0: how automated build workflows, installer packaging (DMG/NSIS/AppImage/deb), and cross-repo automation (Homebrew tap push, WinGet manifest submission) integrate with the existing StorCat Go/Wails architecture. The existing v2.1.0 source and CLI architecture are unchanged — this is entirely additive CI/CD infrastructure.

## Current Architecture (v2.1.0 Baseline)

```
Repos (3):
  scottkw/storcat              main source, .github/workflows/build.yml
  scottkw/winget-storcat       winget manifests (manual)
  scottkw/homebrew-storcat     Casks/storcat.rb (manual)

Current CI (build.yml — triggers on push/PR):
  build-macos   → wails build -clean -platform darwin/universal → artifact: StorCat.app
  build-windows → wails build -clean -platform windows/amd64   → artifact: StorCat.exe
  build-linux   → wails build -clean -platform linux/{amd64,arm64} → artifact: StorCat

Release process (fully manual):
  1. Developer creates GitHub release
  2. Developer runs update-tap.sh → pushes new cask to homebrew-storcat
  3. Developer runs update-winget.sh → creates YAML manifests locally
  4. Developer submits winget manifests to microsoft/winget-pkgs by hand
```

## Pattern 1: Wails Build with The-Egg-Corp/wails-build-action

**What:** The most battle-tested Wails-specific CI action (`The-Egg-Corp/wails-build-action@v2`). Handles Go + Node setup, Wails install, build, and optional NSIS installer.

**When to use:** All three platform build jobs.

**Key parameters for StorCat:**
```yaml
- uses: The-Egg-Corp/wails-build-action@v2
  with:
    name: StorCat
    build-name: StorCat          # or StorCat.exe on windows
    build-platform: darwin/universal
    go-version: '1.23'
    node-version: '20'
    nsis: false                  # true for Windows installer job
    package: false               # we handle artifact upload ourselves
```

**NSIS flag:** When `nsis: true`, the action runs `wails build -nsis`. Wails auto-generates `build/windows/installer/` config on first project creation — those files must be present in the repo. Output is `build/bin/StorCat-amd64-installer.exe`.

**Note on `-windowsconsole`:** The action doesn't expose `-windowsconsole` as a parameter. Pass it via a custom build step or override with `wails-build-flags` if the action supports extra flags — otherwise build manually with:
```yaml
run: wails build -clean -platform windows/amd64 -nsis -windowsconsole
```

## Pattern 2: DMG Packaging (macOS)

**What:** After `wails build` produces `build/bin/StorCat.app`, wrap it in a DMG using `L-Super/create-dmg-actions@v1`.

**When to use:** macOS release job only.

```yaml
- name: Create DMG
  uses: L-Super/create-dmg-actions@v1
  with:
    dmg_name: StorCat-${{ env.VERSION }}-macOS
    src_dir: ${{ github.workspace }}/build/bin/StorCat.app
```

**Alternative:** `create-dmg/create-dmg` shell script (install via Homebrew in CI). More control over background image, layout, etc. The action is simpler for CI; the script is better if custom DMG appearance is needed later.

**Output path:** `${{ github.workspace }}/StorCat-{version}-macOS.dmg`

**Code signing:** Not in scope for v2.2.0 (documented as future milestone). The DMG will be unsigned. macOS users will see a Gatekeeper warning on first launch — document this in the release notes as a known limitation until the signing milestone.

## Pattern 3: Linux AppImage Packaging

**What:** After `wails build` produces `build/bin/StorCat`, create a portable AppImage using `appimagetool`.

**When to use:** Linux release jobs.

**Approach — AppDir + appimagetool (no extra action needed):**
```yaml
- name: Create AppImage
  run: |
    # Create AppDir structure
    mkdir -p AppDir/usr/bin AppDir/usr/share/applications AppDir/usr/share/icons/hicolor/256x256/apps
    cp build/bin/StorCat AppDir/usr/bin/storcat
    # Copy icon and desktop file from build/linux/
    cp build/linux/icon.png AppDir/usr/share/icons/hicolor/256x256/apps/storcat.png
    # AppImage requires a desktop file and AppRun symlink at root
    cat > AppDir/storcat.desktop << 'EOF'
    [Desktop Entry]
    Name=StorCat
    Exec=storcat
    Icon=storcat
    Type=Application
    Categories=Utility;
    EOF
    ln -s usr/bin/storcat AppDir/AppRun
    chmod +x AppDir/AppRun
    # Download appimagetool and create AppImage
    wget -qO appimagetool https://github.com/AppImage/AppImageKit/releases/download/continuous/appimagetool-x86_64.AppImage
    chmod +x appimagetool
    ARCH=x86_64 ./appimagetool AppDir StorCat-${{ env.VERSION }}-Linux-x64.AppImage
```

**ARM64 note:** `appimagetool` has separate ARM64 binary (`appimagetool-aarch64.AppImage`). Must use the correct architecture binary when building for arm64.

## Pattern 4: Linux deb Packaging

**What:** Create a `.deb` package from the StorCat binary for Debian/Ubuntu users.

**When to use:** Linux amd64 release job (arm64 optional for v2.2.0).

**Approach — manual dpkg-deb (no external action needed):**
```yaml
- name: Create deb package
  run: |
    VERSION="${{ env.VERSION }}"  # e.g. 2.2.0
    PKGDIR="storcat_${VERSION}_amd64"
    mkdir -p "${PKGDIR}/DEBIAN"
    mkdir -p "${PKGDIR}/usr/bin"
    mkdir -p "${PKGDIR}/usr/share/applications"
    cp build/bin/StorCat "${PKGDIR}/usr/bin/storcat"
    cat > "${PKGDIR}/DEBIAN/control" << EOF
    Package: storcat
    Version: ${VERSION}
    Architecture: amd64
    Maintainer: Ken Scott <kenscott@gmail.com>
    Description: StorCat - Storage Media Cataloging Tool
    Homepage: https://github.com/scottkw/storcat
    EOF
    dpkg-deb --build "${PKGDIR}" "storcat_${VERSION}_amd64.deb"
```

**dpkg-deb** is pre-installed on GitHub's ubuntu-latest runners — no additional tooling needed.

## Pattern 5: Release Asset Publishing with softprops/action-gh-release

**What:** Collect all artifacts from parallel build jobs and publish them to a GitHub release in a single job.

**When to use:** `create-release` job that runs after all build jobs complete.

```yaml
create-release:
  needs: [build-macos, build-windows, build-linux-amd64, build-linux-arm64]
  runs-on: ubuntu-latest
  steps:
    - uses: actions/download-artifact@v4
      with:
        path: artifacts/
        merge-multiple: true
    - name: Generate SHA256 checksums
      run: |
        cd artifacts && sha256sum * > SHA256SUMS.txt
    - uses: softprops/action-gh-release@v2
      with:
        files: |
          artifacts/*
        generate_release_notes: true
```

**Trigger guard:** The release workflow should only run on version tags, not all tags:
```yaml
on:
  push:
    tags:
      - 'v[0-9]+.[0-9]+.[0-9]+'
```

## Pattern 6: Homebrew Tap Auto-Update

**What:** After GitHub release is created, download the macOS DMG, compute SHA256, render the cask template, and push to `scottkw/homebrew-storcat`.

**When to use:** `update-homebrew` job, triggered after `create-release`.

**Required secret:** `HOMEBREW_TAP_TOKEN` — classic PAT with `repo` scope on `scottkw/homebrew-storcat`.

```yaml
update-homebrew:
  needs: create-release
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v4

    - name: Render cask from template
      env:
        VERSION: ${{ github.ref_name }}  # e.g. v2.2.0
      run: |
        VERSION_NO_V="${VERSION#v}"
        DMG_URL="https://github.com/scottkw/storcat/releases/download/${VERSION}/StorCat-${VERSION_NO_V}-macOS.dmg"
        SHA256=$(curl -sL "${DMG_URL}" | sha256sum | awk '{print $1}')
        sed -e "s/{{VERSION}}/${VERSION_NO_V}/g" \
            -e "s/{{SHA256_DMG}}/${SHA256}/g" \
            packaging/homebrew/storcat.rb.template > /tmp/storcat.rb

    - name: Push cask to homebrew-storcat
      env:
        HOMEBREW_TAP_TOKEN: ${{ secrets.HOMEBREW_TAP_TOKEN }}
      run: |
        git clone "https://x-access-token:${HOMEBREW_TAP_TOKEN}@github.com/scottkw/homebrew-storcat.git" tap
        cp /tmp/storcat.rb tap/Casks/storcat.rb
        cd tap
        git config user.email "github-actions[bot]@users.noreply.github.com"
        git config user.name "github-actions[bot]"
        git add Casks/storcat.rb
        git commit -m "Update StorCat to ${VERSION}"
        git push
```

**Alternative:** `Homebrew Releaser` action from the Marketplace. More opinionated — useful if you want automated PR review. The manual push approach above is simpler and gives full control.

## Pattern 7: WinGet Manifest Auto-Submission

**What:** After GitHub release is created, automatically submit a PR to `microsoft/winget-pkgs` with updated manifests.

**When to use:** `update-winget` job, triggered after `create-release`.

**Required secret:** `WINGET_TOKEN` — classic PAT with `public_repo` scope.

**Prerequisite:** Package must already be listed in `microsoft/winget-pkgs` under `scottkw.StorCat`. The first submission must be manual; subsequent releases are automated.

```yaml
update-winget:
  needs: create-release
  runs-on: ubuntu-latest
  steps:
    - uses: vedantmgoyal9/winget-releaser@main
      with:
        identifier: scottkw.StorCat
        max-versions-to-keep: 5
        token: ${{ secrets.WINGET_TOKEN }}
```

**What it does internally:** Downloads the Windows installer from the GitHub release, computes SHA256, generates 3 YAML manifest files, forks `microsoft/winget-pkgs` to the token owner's account (or reuses existing fork), and opens a PR. Uses Komac internally.

**Constraint:** `installers-regex` defaults to `.(exe|msi|msix|appx)(bundle){0,1}$` — it will pick up the NSIS installer automatically if named `.exe`.

## Integration Points

### Existing → New

| Existing Component | Integrates With | Integration Type |
|--------------------|-----------------|-----------------|
| `wails.json` (productVersion) | `release.yml` env.VERSION | Version read from tag (`github.ref_name`), not wails.json — must stay in sync |
| `build/windows/installer/` | `wails build -nsis` | NSIS config files must be committed — auto-generated by Wails project creation, verify they exist |
| `build/bin/StorCat.app` | `create-dmg` action | Direct path dependency — action reads from `build/bin/` |
| `build/bin/StorCat.exe` | Release upload | Direct — rename to include version in upload step |
| `build/bin/StorCat` (Linux) | AppImage + deb steps | Direct — binary path known |

### New → External

| New Component | External Target | Authentication |
|---------------|-----------------|----------------|
| `release.yml` update-homebrew | `scottkw/homebrew-storcat` | `HOMEBREW_TAP_TOKEN` secret (PAT, repo scope) |
| `release.yml` update-winget | `microsoft/winget-pkgs` | `WINGET_TOKEN` secret (classic PAT, public_repo scope) |
| `release.yml` create-release | `scottkw/storcat` releases | `GITHUB_TOKEN` (automatic, no secret needed) |

### Build.yml vs Release.yml — Trigger Separation

The existing `build.yml` triggers on `push` to branches and PRs — keep this unchanged for CI. The new `release.yml` triggers only on version tags (`v*.*.*`). They are independent workflows and should remain separate files. Do not merge them into one workflow — the complexity and conditional logic is not worth it.

```
build.yml  → triggers on: push (branches), pull_request
           → purpose: verify builds compile on every commit
           → does NOT create releases or packages

release.yml → triggers on: push (tags matching v*.*.*)
            → purpose: build, package, publish, update satellites
            → full pipeline including packaging and distribution
```

## Scalability Considerations

This is CI/CD infrastructure, not user-facing scale. The relevant dimension is build time.

| Concern | Current | After v2.2.0 |
|---------|---------|--------------|
| Build time (per platform) | ~5-8 min (wails install + npm + build) | ~8-12 min (add packaging steps) |
| Total release time | Manual, unbounded | ~20-25 min parallel + 5 min release job |
| GitHub Actions minutes | Free tier may be sufficient | macOS runners cost 10x; monitor usage |
| Windows cross-compile from macOS | Not viable (no WebView2) | Build on windows-latest runners |

**macOS runner cost note:** GitHub Actions charges 10x rate for macOS runners vs Linux. The existing `build.yml` already runs `build-windows` on `macos-latest` (cross-compiling). The release workflow should move Windows builds to `windows-latest` — this is the correct approach anyway, as Wails Windows builds work better natively.

## Anti-Patterns to Avoid

### Anti-Pattern 1: Single Mega-Workflow

**What people do:** Combine CI, build, package, and release into one YAML with `if: github.ref_type == 'tag'` conditions throughout.

**Why it's wrong:** The conditional complexity becomes unmanageable. A broken release step can silently skip packaging. Debugging requires reading 200-line YAML with branching logic.

**Do this instead:** Two separate files — `build.yml` (CI) and `release.yml` (release). Each file has a single trigger and a single purpose.

### Anti-Pattern 2: Embedding Version in Source Files

**What people do:** Update `wails.json` `productVersion` or a `VERSION` file as part of the release workflow.

**Why it's wrong:** Creates a circular dependency — the workflow needs to commit back to the repo, which can trigger another workflow run. Version state lives in two places (git tag + source file), which must be kept manually in sync.

**Do this instead:** Use the git tag as the single source of truth for release version. Extract `VERSION="${{ github.ref_name }}"` in the workflow. The `wails.json` productVersion is used only for local builds; it does not gate the release.

### Anti-Pattern 3: Building Windows on macOS Runners (Cross-Compile)

**What people do:** The existing `build.yml` builds Windows on `macos-latest` using Wails cross-compilation.

**Why this is a problem for installers:** NSIS (`wails build -nsis`) requires NSIS to be installed, which works on Windows runners but is less predictable on macOS. The `-windowsconsole` flag behavior is also harder to validate without a native Windows runner.

**Do this instead:** Move Windows release builds to `windows-latest`. Cross-compilation was acceptable for CI (raw binary check); it's not acceptable when you need NSIS installer generation.

### Anti-Pattern 4: Blocking Release on WinGet PR Merge

**What people do:** Wait for `microsoft/winget-pkgs` PR to merge before considering the release "done."

**Why it's wrong:** WinGet PR review can take 24-72 hours. The release workflow should not block on external review timelines.

**Do this instead:** The `update-winget` job submits the PR and exits successfully regardless of merge status. Monitor the PR separately. Homebrew (where you control the tap) updates instantly; WinGet is asynchronous by design.

### Anti-Pattern 5: Storing PAT as an Environment Variable

**What people do:** Hard-code the token in the workflow YAML for simplicity.

**Why it's wrong:** Exposes the token in plaintext in the repository.

**Do this instead:** Store as a GitHub repository secret (`HOMEBREW_TAP_TOKEN`, `WINGET_TOKEN`) and reference via `${{ secrets.TOKEN_NAME }}`. Never commit token values.

## Suggested Build Order for Phases

Dependencies flow: packaging infrastructure must exist before automation can reference it.

### Phase 1: Repo Consolidation (No Automation Yet)

Move packaging artifacts into the main repo. Validate that existing manifests and scripts work from their new location. This is prerequisite work with zero risk to the existing CI.

```
1a. Create packaging/winget/ structure, move manifests from winget-storcat
1b. Create packaging/homebrew/storcat.rb.template from current cask
1c. Move update scripts into packaging/
1d. Verify scripts still work from new paths (local test only)
```

**No workflow changes in Phase 1** — build.yml stays identical.

### Phase 2: Release Workflow — Build and Package

Create `release.yml` with build + packaging jobs only. No satellite repo updates yet. Validates that DMG, NSIS, AppImage, and deb packaging all work before adding cross-repo side effects.

```
2a. Create release.yml with tag trigger
2b. macOS job: build + DMG creation → upload artifact
2c. Windows job: build (windows-latest) + NSIS → upload artifacts
2d. Linux jobs: build + AppImage + deb → upload artifacts
2e. create-release job: collect + publish GitHub release
```

**Validation gate:** Tag a test release (e.g. `v2.2.0-rc1`), verify all 7 expected assets appear in the GitHub release.

### Phase 3: Satellite Updates

Add the `update-homebrew` and `update-winget` jobs to `release.yml`. These run after `create-release` and have external side effects.

```
3a. Set up HOMEBREW_TAP_TOKEN and WINGET_TOKEN secrets
3b. Add update-homebrew job — test with a release candidate tag
3c. Add update-winget job — requires package already in winget-pkgs
3d. Verify homebrew-storcat Casks/storcat.rb is updated correctly
3e. Verify WinGet PR is opened on microsoft/winget-pkgs
```

### Phase 4: Cleanup

```
4a. Archive scottkw/winget-storcat (mark as archived on GitHub)
4b. Update homebrew-storcat README: "auto-managed, do not edit manually"
4c. Update storcat README: document new install methods
```
