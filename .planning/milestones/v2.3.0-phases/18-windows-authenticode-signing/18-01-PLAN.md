Modify the `build-windows` job in `.github/workflows/release.yml`. Three changes:

**Change 1: Add `environment: release` to the build-windows job definition.**
Currently line 120-121 reads:
```yaml
  build-windows:
    runs-on: windows-2022
    steps:
```
Change to:
```yaml
  build-windows:
    runs-on: windows-2022
    environment: release
    steps:
```
This exposes ES_USERNAME, ES_PASSWORD, CREDENTIAL_ID, and ES_TOTP_SECRET to the job. Without this line, all secrets are empty and eSigner fails. (Per WSIGN-01, WSIGN-02)

**Change 2: Insert 3 new steps AFTER the "Build Windows with NSIS installer" step and BEFORE the "Rename Windows binary" step.**

Insert these steps in this exact order:

Step A — Sign portable exe:
```yaml
      - name: Sign portable exe
        uses: SSLcom/esigner-codesign@cf5f6c1d38ad10f47e3ed9aca873f429b1a8d85b  # v1.3.2
        with:
          command: sign
          username: ${{ secrets.ES_USERNAME }}
          password: ${{ secrets.ES_PASSWORD }}
          credential_id: ${{ secrets.CREDENTIAL_ID }}
          totp_secret: ${{ secrets.ES_TOTP_SECRET }}
          file_path: build\bin\StorCat.exe
          malware_block: false
          override: true
```
(Per WSIGN-02 — signs portable .exe with Authenticode via SSL.com cloud HSM)

Step B — Sign NSIS installer:
```yaml
      - name: Sign NSIS installer
        uses: SSLcom/esigner-codesign@cf5f6c1d38ad10f47e3ed9aca873f429b1a8d85b  # v1.3.2
        with:
          command: sign
          username: ${{ secrets.ES_USERNAME }}
          password: ${{ secrets.ES_PASSWORD }}
          credential_id: ${{ secrets.CREDENTIAL_ID }}
          totp_secret: ${{ secrets.ES_TOTP_SECRET }}
          file_path: build\bin\StorCat-amd64-installer.exe
          malware_block: false
          override: true
```
(Per WSIGN-01 — signs NSIS installer with Authenticode via SSL.com cloud HSM)

Step C — Verify Authenticode signatures:
```yaml
      - name: Verify Authenticode signatures
        shell: cmd
        run: |
          signtool verify /pa /v build\bin\StorCat.exe
          signtool verify /pa /v build\bin\StorCat-amd64-installer.exe
```
(Per WSIGN-04 — CI gate step; exits non-zero if either signature is invalid)

**Anti-patterns to avoid:**
- Do NOT use `@develop` tag for the eSigner action — use the pinned SHA `cf5f6c1d38ad10f47e3ed9aca873f429b1a8d85b`
- Do NOT sign after rename — eSigner `file_path` must reference Wails canonical output names
- Do NOT use `dlemstra/code-sign-action` — archived October 2025
- Do NOT set `malware_block: true` — paid SSL.com add-on, will cause failures
- Do NOT use `shell: bash` for the signtool verify step — use `shell: cmd`
- Do NOT use `env:` block for secrets in eSigner steps — the action takes them as `with:` inputs directly

**Known limitation (acceptable for v2.3.0):** The installed binary extracted by the NSIS installer in Program Files is unsigned (embedded before post-build signing). The NSIS installer wrapper itself IS signed. This is a Wails limitation (issue #3716). Per WSIGN-01 and WSIGN-02, the requirement is CI signing of the output artifacts, not per-extracted-binary verification.
