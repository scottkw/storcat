### Pattern 1: New Service Method — LoadCatalog
**What:** `LoadCatalog(filePath string) (*models.CatalogItem, error)` reads a catalog JSON file and returns parsed content. The logic is identical to `searchInCatalogFile` but without the search step — read file, try array format, try object format, return root item.
**When to use:** When the frontend needs the full catalog tree (e.g., for display or inspection), not just metadata.
**Example:**
```go
// internal/search/service.go — new method
// LoadCatalog reads and parses a catalog JSON file, returning the root CatalogItem.
// Supports both bare-object format (v2.0.0) and array-wrapped format (v1.0 bash script).
func (s *Service) LoadCatalog(filePath string) (*models.CatalogItem, error) {
    data, err := os.ReadFile(filePath)
    if err != nil {
        return nil, fmt.Errorf("failed to read catalog file: %w", err)
    }

    // Try array format first (v1.0 bash script compatibility)
    var catalogArray []*models.CatalogItem
    if err := json.Unmarshal(data, &catalogArray); err == nil && len(catalogArray) > 0 {
        return catalogArray[0], nil
    }

    // Try bare object format (v2.0.0)
    var catalogObj models.CatalogItem
    if err := json.Unmarshal(data, &catalogObj); err != nil {
        return nil, fmt.Errorf("failed to parse catalog JSON: %w", err)
    }

    return &catalogObj, nil
}
```

**Key insight:** The dual-format parsing logic already exists verbatim in `searchInCatalogFile` (lines 62-73 of `internal/search/service.go`). `LoadCatalog` is a clean extraction of that parsing logic without the search pass-through.

### Pattern 2: App Method Wrapper — App.LoadCatalog
**What:** `App.LoadCatalog(filePath string) (*models.CatalogItem, error)` in `app.go` delegates to `searchService.LoadCatalog`. Follows the same pattern as every other `App` method — resolve to absolute path, delegate, return.
**When to use:** Every Go method exposed to the frontend must go through `app.go`.
**Example:**
```go
// app.go — new method
// LoadCatalog reads and parses a catalog JSON file
func (a *App) LoadCatalog(filePath string) (*models.CatalogItem, error) {
    absPath, err := filepath.Abs(filePath)
    if err != nil {
        return nil, err
    }
    return a.searchService.LoadCatalog(absPath)
}
```

### Pattern 3: wailsAPI.ts Wrapper — loadCatalog
**What:** A new `loadCatalog` entry in the `wailsAPI` object in `frontend/src/services/wailsAPI.ts`. It imports `LoadCatalog` from the Wails binding and wraps the result in a `{success, catalog}` envelope to match the Electron `load-catalog` contract exactly.
**When to use:** All Wails Go calls go through `wailsAPI` to maintain the `window.electronAPI` interface contract.

**Electron v1 reference (confirmed from git commit `9c38abb1`):**
```javascript
// src/main.js Electron handler
ipcMain.handle('load-catalog', async (event, filePath) => {
  try {
    const content = await fs.readFile(filePath, 'utf8');
    const catalog = JSON.parse(content);
    return { success: true, catalog };
  } catch (error) {
    return { success: false, error: error.message };
  }
});
```

**Go/Wails equivalent wrapper:**
```typescript
// frontend/src/services/wailsAPI.ts — new entry in wailsAPI object
loadCatalog: async (filePath: string) => {
  try {
    const catalog = await LoadCatalog(filePath);
    return { success: true, catalog };
  } catch (error: any) {
    return { success: false, error: error.message || 'Unknown error' };
  }
},
```

The import at the top of `wailsAPI.ts` must add `LoadCatalog` to the destructured import from `'../../wailsjs/go/main/App'`.

### Anti-Patterns to Avoid
- **Implementing LoadCatalog only in app.go with inline logic:** Breaks the pattern where business logic lives in service layer. The method belongs in `search.Service` because it's catalog data access, not application lifecycle.
- **Skipping `wails generate module` after adding `App.LoadCatalog`:** The TypeScript binding won't include the new function. Import in `wailsAPI.ts` will fail at compile time.
- **Implementing LoadCatalog only once (not dual-format):** v1.0 catalogs use array-wrapped format. The existing `searchInCatalogFile` already handles this — copy the same dual-format logic.
- **Re-implementing file reading from scratch:** The `searchInCatalogFile` method already contains the exact parsing logic needed. Extract it, don't rewrite it.

## Common Pitfalls

### Pitfall 1: Stale Wails Bindings After app.go Change
**What goes wrong:** `wailsAPI.ts` imports `LoadCatalog` from `'../../wailsjs/go/main/App'`, but the binding file doesn't export `LoadCatalog` yet. TypeScript compilation fails.
**Why it happens:** Wails generates bindings at build time or when `wails generate module` is run. Simply editing `app.go` does not regenerate them.
**How to avoid:** Add `wails generate module` as an explicit step immediately after modifying `app.go`. Verify `App.d.ts` contains `LoadCatalog` before writing `wailsAPI.ts`.
**Warning signs:** TypeScript error `Module '"../../wailsjs/go/main/App"' has no exported member 'LoadCatalog'`.

### Pitfall 2: Single-Format JSON Parsing
**What goes wrong:** Implementing `LoadCatalog` to parse only bare-object format breaks when given a v1.0 catalog file (array-wrapped `[{...}]`).
**Why it happens:** v1.0 catalogs from the bash script `sdcat.sh` use array format. The backward compatibility requirement is explicit.
**How to avoid:** Copy the dual-format try-array-first, try-object-second logic from `searchInCatalogFile`. Do not simplify to `json.Unmarshal` into `models.CatalogItem` only.
**Warning signs:** `LoadCatalog` returns a parse error for old catalog files.

### Pitfall 3: Missing `loadCatalog` in `window.electronAPI` Type Declaration
**What goes wrong:** `wailsAPI.ts` adds `loadCatalog` to the `wailsAPI` object but the `typeof wailsAPI` declaration is inferred — TypeScript will see it. However, if a frontend component calls `window.electronAPI.loadCatalog`, TypeScript may complain if the method isn't present at the declaration time.
**Why it happens:** The `Window` interface is declared as `electronAPI: typeof wailsAPI`, so adding to `wailsAPI` automatically extends the type. This pitfall is less likely given the existing pattern, but verify.
**How to avoid:** After adding `loadCatalog` to `wailsAPI`, confirm `tsc --noEmit` passes.

### Pitfall 4: DATA-03/04/05 Verification Without the Size Column
**What goes wrong:** The BrowseCatalogsTab does not currently render a `size` column (only `title`, `name`, `modified` in the column definitions at lines 111-152). The Phase 2 success criterion says "Browse tab displays a size column with byte values." This requires a frontend change in addition to the backend fix.
**Why it happens:** Phase 1 added `size` to the Go struct and fixed the Wails binding. Phase 2 success criteria require the column to actually appear in the UI.
**How to avoid:** Phase 2 plan must include a task to add the `size` column to `BrowseCatalogsTab.tsx` column definitions, with a byte-formatting render function.
**Warning signs:** Go tests pass and binding is correct, but the Browse tab table shows no size column.
